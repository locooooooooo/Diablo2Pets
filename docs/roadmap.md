# 暗黑 2 桌宠助手上线级迭代计划

## 1. 北极星目标

把当前项目从“可运行的桌面工具”推进到“可长期常驻、可交付给别人直接使用、UI/UX 和功能都完整闭环的桌宠产品”。

最终产品定位不变，但标准要升级：

- 它首先是一个陪刷桌宠，负责状态感知、节奏记录、情绪反馈和快捷流转。
- 它其次是一个战利品账本，负责截图、OCR、记录、筛选、日报和回看。
- 它最后才是一个赫拉迪姆工坊，负责符文 / 宝石 / 金币自动化和 profile 运维。

一句话目标：

> 做成一个可以打包交付、打开就能用、能长期挂在桌面上的暗黑 2 陪刷助手。

---

## 2. 当前基线

### 已经完成的能力

- Electron 桌宠壳，支持透明、无边框、托盘、全局热键。
- 面板态 / 悬浮态双模式。
- 每日刷图计数、单次耗时、今日统计。
- 掉落记录、`Ctrl+V` 截图保存、OCR 预识别。
- 符文 / 宝石 / 金币三条自动化 runtime。
- Profile 录制、旧配置导入、执行日志查看。
- 开机自启、系统通知、Windows exe 打包。

### 当前主要短板

- UI 视觉已经初步成型，但还没有完全达到“上线级一致性”。
- 高优主流程可用，但仍缺少更多引导、纠错和异常兜底。
- 自动化链路能跑，但联调反馈、预检和错误解释还不够清晰。
- 数据层目前以本地 JSON 为主，报表、角色维度、历史分析还不完整。
- 打包产物仍依赖本机 Python，离“真正独立交付”还有距离。
- 缺少系统化的回归验证、发布清单和 Beta 验收标准。

---

## 3. 迭代原则

### 原则 1：先打磨高频主流程，再补低频高级能力

优先保证用户每天都会碰到的四个动作顺滑：

- 打开 / 呼出桌宠
- 开始 / 结束刷图
- 记录掉落
- 运行常用自动化任务

### 原则 2：每一批都必须“能演示、能回归、能交付”

每个批次结束时都至少要满足：

- 有明确可见的交付物
- 有一组能执行的验收标准
- 不破坏上一批的核心能力

### 原则 3：UI/UX 重构不能脱离真实功能

这不是单独做一层“好看皮肤”，而是要让 UI、流程、反馈、错误处理和发布能力一起升级。

### 原则 4：桌宠感和生产力感要同时成立

它要像 QQ 宠物一样常驻、轻盈、会反馈；
也要像一个真正的工具产品一样稳定、可追溯、可配置。

---

## 4. 分批次路线图

## Batch 0：基线冻结与质量地基

### 目标

先把现有版本的能力边界、数据结构、关键流程和回归清单固定下来，避免后续 UI 大改时把功能打散。

### 交付内容

- 梳理现有功能矩阵和数据流。
- 固定本地数据结构版本。
- 增加关键流程的 smoke checklist。
- 整理 UI 文案、状态、空态、错误态清单。
- 明确 Windows 打包、运行、日志目录和调试路径。

### 验收标准

- 新人拿到项目后能按文档跑起 dev / build / dist。
- 至少有一份“刷图 / 掉落 / 工坊”三大主流程的手工回归表。
- 改 UI 时不再需要靠临时记忆判断有没有改坏功能。

### 风险与依赖

- 如果这一步跳过，后续每次改 UI 都容易把自动化和掉落链路带坏。

---

## Batch 1：发布级 UI 壳层与设计系统

### 目标

把现在的“能看”提升到“统一、稳定、可持续扩展”的 UI 系统层级。

### 交付内容

- 建立统一的颜色 token、字体层级、阴影、圆角、边框和间距系统。
- 收敛按钮、输入框、卡片、标签、toast、空态、加载态组件语言。
- 统一面板态 / 悬浮态 / 托盘态的视觉关系。
- 统一首页、战报、工坊三页的标题系统和导航逻辑。
- 补齐可访问性底线：
  - 44px 触达尺寸
  - 键盘可操作
  - 焦点样式可见
  - 对比度达标

### 验收标准

- 三个页面看起来属于同一产品，而不是三个风格拼接。
- 窄窗口下信息不拥挤，不出现横向溢出。
- 常用按钮一眼能识别主次，危险操作不混淆。

### 不在本批做

- 不新增复杂业务逻辑。
- 不提前做“重动画炫技”。

---

## Batch 2：高频主流程打磨

### 目标

把“陪刷首页”和“战利品账本”打磨到真正顺手。

### 交付内容

#### 陪刷首页

- 快速开始 / 结束刷图的单步交互。
- 地图模板与最近路线建议。
- 今日节奏卡片：
  - 次数
  - 总耗时
  - 平均单次
  - 最近一次
- 最近战果预览和快捷跳转。

#### 战利品账本

- 掉落记录表单进一步精简。
- OCR 结果可编辑、可一键采用、可一键忽略。
- 稀有词高亮词典和分类纠错。
- 战报筛选增强：
  - 关键词
  - 类别
  - 场景
  - 高亮
- 空态、无 OCR、OCR 失败、截图无效等反馈补齐。

### 验收标准

- 用户在 3 步内完成一次掉落记录。
- OCR 失败时不会卡住主流程。
- 首页打开后 3 秒内能判断“现在刷了多少、刚刚掉了什么、下一步去哪”。

---

## Batch 3：赫拉迪姆工坊闭环

### 目标

把工坊从“能运行”提升到“能联调、能诊断、能放心用”。

### 交付内容

- 三类任务统一成一屏可理解的操作模型：
  - 输入
  - 试运行
  - 正式执行
  - Profile
  - 日志
- 增加任务预检：
  - Python 是否存在
  - profile 是否存在
  - 必填参数是否为空
  - 截图路径是否有效
- 增加执行中反馈：
  - 当前阶段
  - 运行中提示
  - 成功 / 失败摘要
- 增加日志解析摘要，不只展示原始日志全文。
- 宝石截图识别补一层结果确认。
- 金币任务明确“预计批次”和风险提示。

### 验收标准

- 用户不看 README 也能从工坊界面完成一次完整联调。
- 常见失败都能在界面里给出可理解原因。
- 试运行和执行的区别足够清晰，不会误触。

---

## Batch 4：数据层与报表升级

### 目标

让产品从“记录今天”升级到“能看趋势、能沉淀资产”。

### 交付内容

- 周报 / 月报 / 场景收益报表。
- 地图维度统计和收益排行。
- 多角色数据隔离与切换。
- 历史筛选和时间范围查询。
- 稀有掉落卡片化展示。
- 导出能力：
  - 导出图片
  - 导出 PDF
  - 导出 JSON/CSV

### 验收标准

- 能回答“这周主要刷了什么、哪里收益更好、掉了哪些高亮物”。
- 用户不需要翻 `data.json` 就能回看历史成果。

---

## Batch 5：桌宠人格与陪伴感

### 目标

把“工具感”继续升级成“陪伴感”，让它更像桌宠，而不是悬浮小面板。

### 交付内容

- 待机动作、呼吸感、轻量入场动效。
- 情绪状态机：
  - 待命
  - 陪刷中
  - 刚出高亮掉落
  - 长时间未记录
- 气泡脚本系统：
  - 今日状态
  - 稀有掉落反馈
  - 工坊执行结果反馈
- 托盘 / 悬浮态联动反馈。
- 事件驱动的小惊喜设计，但不打扰主流程。

### 验收标准

- 悬浮态在桌面上有“宠物感”，不是单纯小窗压缩版。
- 所有动效都服务于状态表达，不影响操作。
- 支持 reduced motion 降级。

---

## Batch 6：发布工程与独立交付

### 目标

让它真正具备“发给别人就能跑”的交付能力。

### 交付内容

- 内嵌 Python 或无 Python 依赖方案。
- 安装包 / 绿色版双交付策略。
- 首次启动引导：
  - 数据目录说明
  - 自动化依赖检查
  - profile 初始化
- 崩溃日志、运行日志、问题导出。
- 发布前 smoke test。
- 版本号、更新日志、发布包命名规范。

### 验收标准

- 新机器上无需手配 Python 即可运行核心能力。
- 出问题后能快速拿到日志定位。
- 每次发版都有固定 checklist 和回滚方案。

---

## Batch 7：Beta 验收与 v1.0

### 目标

在真实使用场景里跑一轮 Beta，把产品磨到能正式对外给人用。

### 交付内容

- 真实玩家场景 Beta 清单。
- 一轮 bug bash。
- UI/UX 细节打磨收口。
- 自动化任务实战校准。
- v1.0 发布说明和演示截图。

### v1.0 验收标准

- 桌宠壳稳定，不白屏、不丢状态、不莫名消失。
- 陪刷首页、战报、工坊三条主线完整闭环。
- 打包版可直接运行，关键依赖可检查。
- Beta 使用者能在不看源码的前提下完成核心流程。

---

## 5. 建议执行顺序

如果按最稳的方式推进，推荐下面这个顺序：

1. `Batch 0` 基线冻结
2. `Batch 1` UI 壳层与设计系统
3. `Batch 2` 高频主流程
4. `Batch 3` 工坊闭环
5. `Batch 6` 发布工程
6. `Batch 4` 数据报表
7. `Batch 5` 桌宠人格
8. `Batch 7` Beta 与 v1.0

这样安排的原因是：

- 先把产品壳和主流程做稳，避免“炫但不好用”。
- 先把发布能力做起来，避免功能越来越多但别人用不了。
- 桌宠人格和高级报表可以在稳定基础上继续叠加。

---

## 6. 下一批建议直接开做什么

下一批建议正式进入：

## Sprint A：`Batch 0 + Batch 1`

也就是先完成两件事：

- 把当前版本整理成可回归、可验收的基线。
- 把 UI 系统层和三页视觉一致性打透。

### Sprint A 的明确交付

- 一份功能 / 页面 / 状态清单
- 一份手工回归清单
- 一套统一视觉 token
- 首页 / 战报 / 工坊三页统一重构
- 悬浮态与面板态视觉关系重构

### Sprint A 完成后，再进入 Sprint B

`Sprint B = Batch 2 + Batch 3`

也就是把“掉落记录”和“工坊联调”两条高频使用链路打磨到真正顺手。

---

## 7. 结论

后续不再按“有空想到什么就加什么”的方式推进，而是按下面这条主线执行：

> 先做稳定的 UI 壳和主流程，再做工坊闭环和发布能力，最后再把它打磨成真正有陪伴感的桌宠。

这样做的结果是：

- UI/UX 会越来越像上线产品，而不是 demo。
- 功能会越来越完整，但不会越做越乱。
- 每一批都能打包、验证、继续迭代。
