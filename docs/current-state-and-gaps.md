## 2. 当前最严重的问题

### P0：3C 自动计数没有真正闭环

这是目前最影响可用性的阻塞项。

本地数据状态显示：

- `2026-04-10` 当前 `counterSession` 仍停在 `waiting-for-game`
- `lastDetectedState` 是 `unknown`
- `completedRuns` 还是 `0`
- 没有生成新的 `runHistory`

也就是说：

- 会话启动了
- 监控也在跑
- 但状态识别没有成功进入“大厅”或“游戏内”
- 所以没有真正开始计时，也没有结算每一把

### 直接原因

当前识别结果里，大厅模板只打到了约 `0.685`，没有达到脚本设定的模板阈值。

这说明当前不是“逻辑没写”，而是“识别没命中”。

### 高概率原因

1. 现有模板和你的游戏实际分辨率 / 缩放 / 界面状态不一致
2. 模板阈值偏高
3. 当前模板选取点不够稳定
4. 状态识别仍然过度依赖固定模板，没有做足够的容错

## 3. 产品方向为什么跑偏了

现在和你心目中的应用差距大，核心不是“功能少”，而是“重心偏了”。

### 当前偏差

- 首页承载了太多成长、收藏、世界观表达
- 桌宠人格演出抢了主流程注意力
- 收藏册、房间、Atlas 这类内容投入过早
- 工坊虽然功能多，但对新手还是偏复杂

### 正确方向

应该先把重心收回：

- 陪刷计数是否可靠
- 掉落记录是否顺手
- 工坊是否闭环
- 桌宠是否轻量常驻

## 4. 当前技术债

### 编码污染

仓库里已有多份文档和一部分界面文案出现乱码或编码污染。

影响：

- 状态提示不好理解
- 错误信息不可信
- 文档难以继续维护

### 工作区历史残留多

当前仓库存在不少旧调试文件、历史试验文件和未清理分支产物。

影响：

- 容易误引用错误文件
- 打包和调试时更难判断问题来源

### 识别链路缺乏观测

自动计数当前最缺的是“识别过程可观测”：

- 当前命中了什么模板
- 置信度多少
- 为什么没过阈值
- 当前屏幕截图是什么

如果没有这层观测，就很难快速把 3C 自动计数调稳。

## 5. 现在应该先做什么，不该做什么

### 现在先做

1. 让 3C 自动计数真正跑通
2. 重建文档基线
3. 重构页面主路径
4. 清理乱码和关键状态文案

### 现在先不做

1. 不继续扩收藏系统
2. 不继续扩成长线
3. 不继续扩剧情演出
4. 不继续做复杂宠物世界观

## 6. 结论

当前最合理的判断是：

- 这套工程已经具备不错的桌面壳和功能基础
- 但还没有进入“可长期使用”的产品阶段
- 当前必须回到“主流程稳定 + 方向重置 + 文档统一”三件事

其中第一优先级是：

`先把 3C 自动计数调到真实可用`
